## Context

当前 `qmt-proxy` 是一个以 FastAPI 为核心的后端服务，已经具备以下前端可直接复用的能力：

- `POST /api/v1/data/subscription`、`GET /api/v1/data/subscriptions`、`GET /api/v1/data/subscription/{subscription_id}`、`DELETE /api/v1/data/subscription/{subscription_id}` 用于订阅生命周期管理。
- `GET /ws/quote/{subscription_id}` 用于推送订阅对应的实时行情消息，并支持 `ping` / `pong` 心跳。
- `POST /api/v1/data/market` 以及其他数据路由可作为查询面板的数据源。

仓库目前没有前端工程，也没有静态资源托管入口。要实现可用的 Web UI，需要同时覆盖前端项目初始化、后端静态资源挂载、REST 与 WebSocket 的统一接入方式、以及对现有认证方式的前端封装。

约束如下：

- 不能破坏现有 API 与 WebSocket 路由的兼容性。
- UI 首版应直接基于现有后端契约实现，不要求新增后端业务接口。
- 当前鉴权实际依赖 `Authorization: Bearer <api-key>`，UI 需要与这一行为保持一致。
- 行情推送是持续流，前端必须限制内存占用，不能无限累积消息。

## Goals / Non-Goals

**Goals:**

- 新增一个基于 React、TypeScript、Ant Design 的前端工作台。
- 在一个统一界面里完成 API Key 配置、订阅列表查看、订阅创建/取消、实时推送查看和市场数据查询。
- 在生产模式下由 FastAPI 托管前端构建产物，在开发模式下支持前后端分离联调。
- 为后续扩展更多数据面板或交易面板保留清晰的前端目录结构和共享接入层。

**Non-Goals:**

- 不新增新的行情或查询后端接口。
- 不重构现有订阅管理器、WebSocket 协议或认证实现。
- 不在首版中覆盖所有数据服务接口，优先实现市场数据查询和订阅相关视图。
- 不引入复杂的多角色权限体系、服务端会话、或历史持久化存储。

## Decisions

### 1. 以前端独立工作区形式引入 UI，而不是在 FastAPI 模板中直接拼接 HTML

选择在仓库中新增独立前端目录，例如 `web/`，使用 Vite + React + TypeScript + Ant Design 组织页面、构建和测试。

原因：

- 现有内置 `/ws/test` 页面只适合调试，无法支撑复杂的状态管理和多面板交互。
- 独立前端工作区更适合后续扩展查询页面、交易页面和共享组件。
- Vite 的开发代理可以直接转发 `/api` 与 `/ws`，本地联调成本最低。

替代方案：

- 直接在 FastAPI 中返回多个 HTML 模板。该方案初始成本低，但会让状态管理、组件复用和测试迅速失控，因此不采用。

### 2. 生产环境将前端挂载到 `/ui`，保留现有 `/` 根接口行为

前端构建产物使用 FastAPI `StaticFiles` 或等效静态资源机制挂载到 `/ui`，并为前端内部刷新提供 HTML fallback。后端现有 `/` 与 `/info` 等接口保持不变，避免影响现有脚本、SDK 与文档使用习惯。

原因：

- 当前根路径返回服务元信息，直接改成 UI 会造成行为变化。
- `/ui` 作为显式入口，既适合浏览器访问，也方便文档与 README 引导。
- 静态托管与 API 同域后，可避免额外的跨域和部署编排复杂度。

替代方案：

- 将 UI 挂载到 `/` 并迁移根接口。该方案对现有使用者存在兼容性风险，因此不采用。
- 完全独立部署前端。该方案可行，但会增加环境配置和交付复杂度，因此首版不作为默认方案。

### 3. 使用“单工作台 + 分区面板”而不是多页面复杂路由

首版 UI 采用单页工作台布局，顶部提供连接配置，主体使用 Ant Design `Tabs` 或等效分区展示：

- 订阅总览
- 实时推送
- 数据查询

原因：

- 三类核心操作共享同一组连接配置和同一服务上下文。
- 用户主要诉求是联调和观察，不需要复杂的多级导航。
- 单工作台可以减少初始化成本，并降低静态路由与刷新回退处理复杂度。

替代方案：

- 使用多路由页面拆分订阅、行情、查询。该方案在页面数量更多时更合理，但对当前需求过重，因此暂不采用。

### 4. 前端接入层统一封装 REST 与 WebSocket，并在浏览器本地保存连接配置

前端将提供一个共享配置层，保存：

- API 基础地址
- API Key
- 最近一次成功使用的订阅参数和查询参数

配置存储在浏览器 `localStorage`。所有 REST 请求由统一客户端附加 `Authorization: Bearer <api-key>`，WebSocket 地址根据当前基础地址自动推导 `ws://` 或 `wss://`。

原因：

- 连接配置是所有面板的共同前提，集中管理可以避免重复输入。
- 当前后端 API Key 验证行为实际依赖 Bearer Token，统一封装可以减少调用歧义。
- 持久化最近配置可以提升调试效率，但不需要引入服务端配置存储。

替代方案：

- 每个面板单独维护请求配置。该方案会导致重复输入和状态不一致，因此不采用。
- 服务端保存 UI 用户配置。首版没有用户体系，不适合引入。

### 5. 实时推送数据按订阅维度维护有限长度缓冲区

前端为每个已连接的 `subscription_id` 维护独立消息缓冲区，只保留最近 N 条消息（例如 200 条），同时展示最近一条消息的格式化视图与原始 JSON。

原因：

- 行情推送是无限流，若无边界缓存，页面运行时间越长内存越大。
- 订阅维度缓存更符合排错和观察习惯，便于切换不同订阅。
- 同时显示结构化摘要与原始消息，适合处理不同周期和数据字段的变化。

替代方案：

- 将全部消息统一混排。该方案在多个订阅并行时可读性差。
- 只显示最后一条消息。该方案不足以支持排错和短时回看。

### 6. 查询首版聚焦市场数据接口，并使用通用结果渲染策略

查询面板优先对接 `POST /api/v1/data/market`。表单字段映射已有 `MarketDataRequest`，结果区支持：

- 成功时以表格 + 原始 JSON 两种方式查看
- 失败时显示 HTTP 错误信息和后端返回消息

原因：

- 市场数据查询是当前最核心、最稳定且与订阅能力最相关的查询入口。
- 该接口返回结构明确，适合先搭建查询工作台框架。
- 通用渲染模型可以为后续接入财务、板块、指数等接口复用。

替代方案：

- 一次性覆盖全部数据接口。该方案首版工作量大，且容易把设计重心从“可用工作台”扩散为“全量前端产品”，因此不采用。

## Risks / Trade-offs

- [前端新增 Node 工具链] → 需要在 Python 项目中维护额外依赖和构建步骤；通过将前端目录隔离、补充 README 与脚本来降低维护成本。
- [认证文档与实际行为存在偏差] → UI 统一走 `Authorization: Bearer` 并在文档中写明当前实现，避免继续传播 `X-API-Key` 误解。
- [WebSocket 未做单独鉴权] → 首版沿用现状，不在 UI 中伪造不存在的安全保证；在页面说明中提示 WebSocket 依赖订阅 ID 可见性。
- [推送数据字段不稳定] → 结果面板同时保留原始 JSON，避免前端字段映射过窄导致信息丢失。
- [静态挂载可能影响部署] → 将 UI 挂载到独立路径 `/ui`，并保留关闭或回滚到纯 API 服务的能力。

## Migration Plan

1. 新增前端工作区、构建脚本与前端测试。
2. 在开发环境中通过 Vite 代理联通现有 FastAPI `/api` 和 `/ws`。
3. 在 FastAPI 中增加 `/ui` 静态挂载，并在存在前端构建产物时提供入口页面。
4. 更新 README，补充 Web UI 启动方式、默认访问路径和认证说明。
5. 若部署后需要回滚，可先移除 `/ui` 静态挂载或不分发前端构建产物，后端 API 不受影响。

## Open Questions

- 首版查询面板是否只覆盖 `market` 查询，还是同步纳入 `financial`、`sector` 等接口作为可选扩展。
- 是否需要在首版中支持“从列表选择已有订阅后自动连接 WebSocket”，还是先保留显式连接按钮。
- 后续若后端补充 WebSocket 鉴权，是否沿用当前 REST 配置复用同一令牌传递方式。
